# CompliancePatchBench — Self-Improving Compliance Patcher

AI systems can pass tests and still be wrong. CompliancePatchBench trains
agents that must fix code correctly — even when tests are misleading.

Key idea: we train agents in an environment where shortcut fixes are penalized
by hidden constraints.

Production-grade scaffold for an agentic LLM that finds and fixes
compliance / security / code-quality violations in source code.

```
project/
├── __init__.py
├── utils.py              shared logging / JSON parsing / IO helpers
├── task_generator.py     40+ mutated tasks (GDPR, OWASP, multi-file, adversarial)
├── agent.py              robust agent loop (strict JSON + retry + fallback)
├── dataset_builder.py    rollouts → trajectories → filtered SFT dataset
├── train_model.py        LoRA fine-tune pipeline (Unsloth → falls back to peft)
├── rl_trainer.py         RL loop: rollout → reward-to-go advantages → policy update
├── evaluate.py           difficulty-aware benchmark + BEFORE/AFTER comparison
├── hidden_compliance.py  hidden oracle that catches "fixes" passing CI but still leaking PII
├── model_training_.ipynb  main training notebook
├── README.md
└── data/
    ├── tasks.json           ← generated by task_generator
    ├── dataset.jsonl        ← generated by dataset_builder (cheats filtered out)
    ├── trajectories.jsonl   ← full per-step audit log
    ├── trajectories_rl.jsonl ← latest RL transitions (state, action, reward, next_state)
    ├── learning_curve.json  ← reward/success/hidden-violation history per RL iteration
    ├── eval_*.json          ← per-run evaluation reports
    └── lora_adapter/        ← saved LoRA after train_model
```

This package sits *on top of* the existing `environment/` package
(`CompliancePatchEnv`, rules, graders) — it doesn't replace it.

## Quick start (60-second demo, no GPU needed)

```bash
cd CompliancePatchBench
pip install -r requirements.txt        # base deps for the env

# 1. Generate a fresh fleet of 40 tasks
python -m project.task_generator --num 40 --seed 42

# 2. Roll out the heuristic agent and build the SFT dataset
python -m project.dataset_builder --rollouts 1 --min-success 0.5

# 3. Evaluate the baseline (heuristic) on every task
python -m project.evaluate run --tag baseline
```

Expected output (numbers are *not* perfect — see "Why this benchmark is hard to game"):

```
=== EVALUATION SUMMARY ===
  tasks evaluated:        40
  overall success rate:   ~80 %
    easy   success rate:  ~95 %
    medium success rate:  ~65 %
    hard   success rate:  ~60 %
  partial fix rate:        ~8 %
  hidden violation rate:   ~7 %
  cheat resistance:       ~93 %
  status counts:          SUCCESS=32  PARTIAL=3  FAIL=5
```

Performance varies by task difficulty; significant improvement is observed on
complex / adversarial tasks once the model is fine-tuned. The heuristic
backend is **deliberately imperfect** so we can bootstrap an SFT dataset
without paying for an LLM *and* still leave room to demonstrate improvement.

## End-to-end with a real LLM

### Option A — OpenAI / vLLM / Together (any OpenAI-compatible endpoint)

```bash
export OPENAI_API_KEY=sk-...
export API_BASE_URL=https://api.openai.com    # or your vLLM URL

python -m project.dataset_builder \
    --llm openai --model gpt-4o-mini \
    --rollouts 2 --min-success 0.66

python -m project.evaluate run --llm openai --model gpt-4o-mini --tag openai
```

### Option B — LoRA fine-tune (the full self-improvement loop)

Open `project/model_training_.ipynb` in Colab (or run locally on a CUDA box):

1. Install Unsloth + TRL
2. Generate tasks → run rollouts → build SFT dataset
3. LoRA fine-tune Mistral-7B-Instruct (rank=16, alpha=32, dropout=0.05, 2 epochs)
4. Re-evaluate with the LoRA adapter
5. Print the BEFORE/AFTER table

Total wall-clock on a free T4: ~15-20 min. Total cost: $0-$10.

```bash
python -m project.train_model \
    --base-model unsloth/mistral-7b-instruct-v0.3-bnb-4bit \
    --epochs 2 --batch-size 2 --lr 2e-4

python -m project.evaluate run --llm hf --model project/data/lora_adapter --tag lora
python -m project.evaluate compare \
    --before project/data/eval_baseline.json \
    --after  project/data/eval_lora.json

# Optional RL refinement after SFT:
python -m project.rl_trainer \
    --iterations 3 --rollouts-per-task 1 --epochs-per-iteration 1

python -m project.evaluate iterations --metrics project/data/learning_curve.json
```

One-command judge smoke test:

```bash
PYTHONDONTWRITEBYTECODE=1 python -m project.smoke_test
```

This verifies API routes, task generation, RL iteration metrics, and the
`(state, action, reward, next_state, logprob, done)` trajectory schema.

## How the online RL loop works

```
        ┌──────────────────┐
        │ task_generator   │  18 templates (incl. 6 adversarial) × mutations → 40 tasks
        │                  │  (50% easy / 30% medium / 20% hard)
        └────────┬─────────┘
                 │
                 ▼
        ┌──────────────────┐         per-step:
        │ ComplianceAgent  │ ───────► (prompt, completion, action,
        │  (LLM + retry +  │           reward, observation)
        │   fallback)      │
        └────────┬─────────┘
                 │
                 ▼
        ┌──────────────────┐
        │ hidden_compliance│  oracle: re-inspect patched files; catch
        │  (post-rollout)  │  cheats (hash(email), env-default secrets,
        │                  │  partial multi-file fixes …)  →  −0.5 reward
        └────────┬─────────┘
                 │
                 ▼
       trajectories.jsonl  ← full audit log (always written)
                 │
                 ▼
        ┌──────────────────┐
        │ reward filter    │  REJECT hidden_violation; keep success ≥ 0.66
        │ (best-of-N)      │  AND quality ≥ 0.4; dedupe by task_id
        └────────┬─────────┘
                 │
                 ▼
            dataset.jsonl   ← {input, output, messages, text, difficulty, …}
                 │
                 ▼
        ┌──────────────────┐
        │ train_model      │  SFT initialization: LoRA r=16/α=32/dropout=0.05
        │ (Unsloth)        │
        └────────┬─────────┘
                 │
                 ▼
            lora_adapter/   ← starting policy
                 │
                 ▼
        ┌──────────────────┐
        │ rl_trainer       │  repeat N times:
        │                  │  rollout current policy → save trajectories_rl.jsonl
        │                  │  reward-to-go advantages → policy-gradient update
        │                  │  evaluate → learning_curve.json
        └────────┬─────────┘
                 │
                 ▼
        ┌──────────────────┐
        │ evaluate         │  difficulty-aware metrics + cheat-resistance,
        │                  │  compare_iterations for reward/success curves
        └──────────────────┘
```

Every component is independently runnable from the CLI **and** importable
as a Python module so you can plug them into a Prefect / Dagster / Airflow
job for periodic retraining.

The final training pipeline is:

```
heuristic rollouts → SFT initialization → online GRPO rollouts → GRPO-refined policy
```

## RL formulation

CompliancePatchBench now exposes the agent loop as an explicit reinforcement
learning problem:

| RL component | Implementation |
|---|---|
| **State** | Environment observation snapshot: files, violations, CI results, file-read history, current file view, task difficulty, cumulative reward. |
| **Action** | One structured JSON command: `read_file`, `write_patch`, `run_ci`, or `finalize_patch`. |
| **Reward** | Per-step env reward plus terminal penalties for hidden violations, partial fixes, no-fix cases, regressions, and timeouts. CI/test pass, minimal patch quality, bad patch penalties, and hidden constraints all feed the reward. |
| **Episode** | One task rollout until `done=True` or `max_steps`. |
| **Transition** | Stored as `(state, action, reward, next_state, logprob, done)` in `project/data/trajectories_rl.jsonl`. |

`project/rl_trainer.py` uses TRL `GRPOTrainer` for final policy optimization.
Heuristic/tabular rollouts are used only for initial data collection, baseline
comparison, and CPU dry-runs. The final policy update is online GRPO: the
current policy generates JSON actions, `CompliancePatchEnv` executes them,
the reward function returns environment reward, and GRPO updates that same
policy checkpoint for the next iteration.

The RL loop is designed to scale to larger task distributions; this demo uses
a small subset for runtime constraints.

Most importantly: **the agent doesn't just learn to fix code — it learns to
avoid cheating, because the environment penalizes hidden violations.**

Each iteration, the agent explores fixes, receives feedback from a
non-cheatable environment, and updates the same policy to avoid shortcut
solutions.

## Why this RL cannot be cheated

The RL reward is not just "did the regex pass?" It is grounded in three checks:

1. **CI + tests** verify that the patch satisfies visible functional and static checks.
2. **Hidden oracle** penalizes shortcut fixes such as hashed PII, masked PII,
   hardcoded env defaults, weak crypto, and partial multi-file fixes.
3. **Adversarial tasks** deliberately include fake-safe fixes, misleading
   comments, and cross-file dependencies.

That means the learned policy is rewarded for correct fixes and punished for
shortcut behavior. The agent doesn't just learn to fix code — it learns to
avoid cheating, because the environment penalizes hidden violations.

## Robustness & determinism guarantees

| Failure mode                              | Where it's handled                               |
|------------------------------------------|--------------------------------------------------|
| LLM returns prose instead of JSON         | `extract_json` (4 fallback strategies in `utils.py`) |
| LLM uses an unknown `action_type`         | retry up to `max_retries`, then heuristic fallback   |
| LLM never reads the file                  | heuristic still chooses `read_file` first when budget>0 |
| LLM repeats the same patch forever        | `_patches_seen` deduplicates write_patch attempts  |
| Env raises (e.g. invalid line range)      | wrapped in `agent.run` try/except → `TrajectoryResult.error` |
| Episode ends before finalize              | only call `finalize_patch` when `env.state.done == False` |
| Different model temperatures              | `temperature=0.0` everywhere by default           |
| Reproducibility                           | `seed_everything(42)` in `task_generator`         |
| Reviewer smoke test                       | `python -m project.smoke_test` validates API + RL |

## Why this benchmark is hard to game

A patcher that just learns "make the regex shut up" is not what we want.
Three independent layers stop that:

1. **CI + tests** — the env's static analyser still has to be satisfied.
2. **Hidden compliance oracle** (`hidden_compliance.py`) — runs *after* CI on
   the patched codebase looking for shortcut fixes:
   * PII still present in logs even when transformed (`hash(email)`, `b64(email)`, `email[:3]+"***"`)
   * `os.environ.get('SECRET_KEY', 'literal-fallback')` — env-aware but still hardcoded
   * `DEBUG = os.environ.get('DEBUG', True)` — defaults to insecure
   * `name.replace("'","")` masquerading as SQL-injection mitigation
   * MD5/SHA1 used as a password hash
   * Partial multi-file fixes (one IDOR endpoint locked, another left open)
   * `secret_literal_left_behind` — env load below a literal that's still in source
3. **Adversarial tasks** — six templates explicitly invite a cheat. The agent
   only scores well by avoiding the trap *and* fixing the real issue.

The hidden oracle imposes a **−0.5 reward penalty** on any trajectory it
flags, the dataset filter **rejects** flagged trajectories outright, and the
evaluator surfaces a **`cheat_resistance`** metric (`1 − hidden_violations / n`)
plus per-task `SUCCESS / PARTIAL / FAIL` labels so failure modes are
unmistakable. The combined effect is that a model can't reach a high score
just by satisfying the regex grader — it has to fix the underlying problem.

## Dataset schema (`data/dataset.jsonl`)

Each line is one example, ready for SFT (cheat trajectories are dropped):

```json
{
  "task_id":          "sec_hardcoded_secret_v1_bc9485",
  "category":         "security",
  "framework":        ["OWASP"],
  "difficulty":       "easy",
  "adversarial":      false,
  "input":            "TASK: ... INITIAL_OBSERVATION: {...}",
  "output":           "{...action 1...}\n{...action 2...}\n{...action 3...}",
  "messages":         [{"role": "system", ...}, {"role": "user", ...}, ...],
  "text":             "<|system|>\\n...\\n<|user|>\\n...",
  "final_score":      1.6,
  "success_rate":     1.0,
  "quality":          1.0,
  "n_steps":          3,
  "hidden_violation": false,
  "hidden_reason":    "ok"
}
```

`messages` is the chat-format version (drop straight into TRL's `SFTTrainer`
when the tokenizer has a chat template).
`text` is a tokenizer-agnostic flat rendering — useful for trainers that
don't apply a chat template.

## Adding new tasks

1. Open `project/task_generator.py`
2. Add a `_t_my_new_thing(ctx)` builder that returns
   `(codebase_dict, [Violation, ...])`
3. Append a `Template(...)` entry to the `TEMPLATES` registry
4. Re-run `python -m project.task_generator --num 50`

Mutations (rename, severity-shift, padding, red-herrings) are applied
automatically, so a single new template instantly produces several diverse
task variants.

## Notes on the env's verifiability

The existing `CompliancePatchEnv` verifies fixes with regex patterns +
a static-analysis "semantic validator". A rule with **no detection pattern**
(currently `GDPR-ART25`) cannot be marked as fixed by CI — so the bundled
templates target only rules whose fixes the env can verify. If you extend
`environment/rules/` with new patterns, add matching templates and you're
done.
